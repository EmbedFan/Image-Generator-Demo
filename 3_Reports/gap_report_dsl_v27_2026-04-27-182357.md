### G-01 ‚Äî HIGH ‚Äî Negative `scale` has no ¬ß18 error entry or message pattern

**Sections:** ¬ß8.0, ¬ß18.2  
**Gap:** L-N5 (fixed in v2.7) added the prose "negative values are not supported and result in a validation error" to ¬ß8.0. However, ¬ß18.2 contains no corresponding row ‚Äî no message pattern, no severity, no behaviour label. Every other documented error in ¬ß18 carries a message pattern.  
**Suggested fix:** Add a row to ¬ß18.2:

| Condition | Behaviour |
|---|---|
| `scale < 0` | Error halted: `<file>:<line>: error: scale value must be non-negative, got <value>` |

### G-03 ‚Äî HIGH ‚Äî `output-format=jpeg` + `colorspace=RGBA` conflict is undefined

**Sections:** ¬ß6  
**Gap:** JPEG does not support an alpha channel. The document notes that in `output-format=images` mode each RGBA frame is saved as `.png`, but nowhere addresses the case where `output-format=jpeg` is explicitly set and the frame uses `colorspace=RGBA`. The resulting behavior ‚Äî validation error, silent alpha compositing over white, or encoder failure ‚Äî is completely undefined.  
**Suggested fix:** Add a note to ¬ß6:  
*"Specifying `output-format=jpeg` with `colorspace=RGBA` is a validation error: `<file>:<line>: error: jpeg does not support RGBA colorspace`."*  
(Or alternatively: *"Alpha is composited over a white background before JPEG encoding."* ‚Äî choose one and document it.)

### G-04 ‚Äî HIGH ‚Äî Conflicting `output-format` values across frames in a multi-frame script

**Sections:** ¬ß6  
**Gap:** `output-format` is an `<image-param>` declared per frame. In a multi-frame script, different frames could declare different values (e.g., frame 1 says `png`, frame 2 says `gif`). The document never states which value wins, whether it is the first frame, the last, or whether the conflict is an error.  
**Suggested fix:** Add a note to ¬ß6:  
*"In a multi-frame script, `output-format` must be consistent across all frames. If frames declare different values, the first frame's value is used and a warning is emitted for each conflicting frame: `<file>:<line>: warning: output-format '<value>' ignored; using '<first-value>' from first frame`."*

### G-05 ‚Äî HIGH ‚Äî Initial canvas state when no `background()` is called is undefined

**Sections:** ¬ß7  
**Gap:** ¬ß7 says `background()` *"must appear inside a frame body, after the `image` statement"* ‚Äî but does not say it is *mandatory*. If a frame body contains no `background()` call, the starting canvas color or transparency is never defined.  
**Suggested fix:** Add a note to ¬ß7:  
*"If no `background` statement is present, the canvas is initialised as fully transparent for `colorspace=RGBA`, and white (`#FFFFFF`) for `colorspace=RGB` or `colorspace=GRAY`."*  
(Alternatively, make `background()` mandatory and add a validation error.)

### G-06 ‚Äî MEDIUM ‚Äî `corner` / `corner-radius` behavior on `curved` connector is undefined

**Sections:** ¬ß9.2, ¬ß9.4  
**Gap:** ¬ß9.4 states that `corner` applies "at each intermediate vertex." For `connector-type=curved` (Catmull-Rom spline), vertices are smooth control points ‚Äî there are no right-angle turns to bevel or round. Whether `corner` is silently ignored or produces a warning for `curved` connectors is not stated.  
**Suggested fix:** Add a note to ¬ß9.4:  
*"`corner` and `corner-radius` apply only to `straight` and `step` connectors; they are silently ignored when `connector-type=curved`."*

### G-07 ‚Äî MEDIUM ‚Äî Only `start` (without `end`) supplied to `connector` ‚Äî no error entry

**Gap:** ¬ß9.1 states "either `points` or both `start`+`end` must be supplied." The mutual-exclusivity error (both supplied) has an ¬ß18.1 entry. But the case where only `start` is provided without `end` (or vice versa) has no error entry or message pattern.

### G-09 ‚Äî MEDIUM ‚Äî `clip-bounds` and `clip-shape` both present ‚Äî combined clipping undefined

**Sections:** ¬ß12.2  
**Gap:** An object template can declare both `clip-bounds` (a rectangular bounding clip) and `clip-shape` (a geometry-based clip mask). The document does not state what happens when both are present ‚Äî whether they intersect, whether one takes precedence, or whether the combination is an error.  
**Suggested fix:** Add a note to ¬ß12.2:  
*"When both `clip-bounds` and `clip-shape` are specified, the effective clip region is the **intersection** of the two clip areas."*

### G-10 ‚Äî MEDIUM ‚Äî `frame-mode` conflict across frames ‚Äî first-frame rule only in example notes

**Sections:** ¬ß5  
**Gap:** ¬ß17.2 example notes say *"`frame-mode` applies to the first frame and propagates to GIF loop settings."* This first-frame propagation rule is only stated in an example annotation, not in the formal ¬ß5 attribute specification. If two frames declare different `frame-mode` values, the spec provides no normative resolution rule.  
**Suggested fix:** Add a note to ¬ß5:  
*"For multi-frame GIF output, the `frame-mode` of the **first** frame determines the GIF loop setting; `frame-mode` values on subsequent frames are ignored with no warning."*

### G-11 ‚Äî LOW ‚Äî Unary minus not supported in arithmetic expressions ‚Äî never documented

**Sections:** ¬ß13.3 EBNF `<expr-factor>`  
**Gap:** The arithmetic grammar allows `<length>`, `<identifier>`, or `'(' <expr> ')'` inside an `<expr-factor>`. There is no leading `-` (unary negation) alternative. Consequently, expressions such as `pos = (-10, y)` or `scale = -width` inside a function body are syntactically invalid ‚Äî the workaround is `0 - x`. This restriction is never documented.  
**Suggested fix:** Add a note to ¬ß13.3:  
*"Unary negation is not supported in expressions. To negate a value, subtract from zero: `0 - x`."*

---

### G-12 ‚Äî LOW ‚Äî Named color set vague ‚Äî CSS specification level not stated

**Sections:** ¬ß15.1, ¬ß3 EBNF `<named-color>`  
**Gap:** The `<named-color>` EBNF rule lists 20 specific named colors, then appends `(* any other CSS color name recognised by the engine *)`. This makes the valid color set implementation-defined. CSS named color sets differ across levels: CSS Level 1 (16 colors), CSS Level 3 (147 colors), CSS Level 4 (adds `rebeccapurple`). Without specifying the level, conformant implementations may produce different results.  
**Suggested fix:** Replace the comment with:  
*"The engine recognises all **CSS Color Level 3** named colors (147 names, including the 20 listed above)."*
