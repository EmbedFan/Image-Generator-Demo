## Findings

### W-01 — HIGH — `#` token disambiguation (hex color vs comment) — rule not documented

**Sections:** §2.3, §3 EBNF  
**Gap:** §2.3 states that `#` starts a single-line comment extending to the end of the line. The EBNF in §3 also defines hex color as `'#' [0-9A-Fa-f]{6}`. Both uses of `#` appear in the same source text — for example `color=#FF0000`. The tokeniser must distinguish `#` as a **comment start** from `#` as a **hex color prefix** depending on context (whether the parser is currently expecting a `<value>`). This context-sensitive lexing rule is never documented. An implementor has no guidance on how the ambiguity is resolved.  
**Suggested fix:** Add a note to §2.3 (or §3): *"The `#` character is context-sensitive. When the tokeniser is in a value-position context (after `=` in a named parameter or argument list), `#` followed by exactly 6 hexadecimal digits is tokenised as the start of a `<color>` token. In any other position — start of a statement, or `#` not followed by 6 hex digits in value context — `#` begins a single-line comment."*

### W-02 — HIGH — `begin_frame` without `image` statement — no §18.1 error row

**Sections:** §5, §18.1  
**Gap:** The EBNF in §3 makes `<image-def>` a mandatory element of `<frame>` (no `?` or `*` qualifier). If a script contains a `begin_frame` block with no `image` statement, the grammar requires a parse error — but §18.1 has no error row covering this condition. An implementor has no guidance on the message to emit.  
**Suggested fix:** Add to §18.1:

| Error Condition | Message Pattern |
|---|---|
| `begin_frame` body with no `image` statement | `<file>:<line>: error: frame '<name>': missing required 'image' canvas statement` |

---

### W-03 — MEDIUM — `dpi=0` or negative `dpi` — division-by-zero in unit conversion not guarded

**Sections:** §6, §15.3, §18.2  
**Gap:** §15.3 shows that `pt`, `cm`, and `mm` unit conversions all use `DPI` as a divisor or multiplier (e.g., `1pt = DPI / 72 px`). If `dpi=0` is supplied, all three conversions divide by zero at render time. If `dpi` is negative, conversions produce negative pixel lengths. §18.2 has no validation row for `dpi`. The document states the default is `96` but places no constraint on the allowed range.  
**Suggested fix:** Add to §18.2:

| Condition | Behaviour |
|---|---|
| `dpi` value ≤ 0 in `image` canvas statement | Error halted: `<file>:<line>: error: 'dpi' must be greater than zero, got <value>` |

---

### W-04 — MEDIUM — `opacity` out-of-range not covered by §18.2

**Sections:** §7.3, §11, §18.2  
**Gap:** Both `background(src=..., opacity=<0-1>)` (§7.3) and `image(opacity=<0-1>)` (§11) accept `opacity` in range 0.0–1.0. §18.2 added V-05 rows for `RGBA` alpha out-of-range, but the separate `opacity` parameter on the background-image and image primitives is not covered. Supplying `opacity=1.5` or `opacity=-0.2` has no documented error or clamp behaviour.  
**Suggested fix:** Add to §18.2:

| Condition | Behaviour |
|---|---|
| `opacity` value outside 0.0–1.0 | Error halted: `<file>:<line>: error: 'opacity' value '<n>' is out of range 0.0–1.0` |

---

### W-05 — MEDIUM — `output-format=images` with `colorspace=GRAY` — file extension undefined

**Sections:** §6  
**Gap:** The §6 output file naming table states that for `output-format=images`, "RGBA frames always use `.png`". The implication is that all other frames use `.jpeg`. But `colorspace=GRAY` is neither RGBA nor standard RGB/JPEG-native. Whether a GRAY-colorspace frame in `images` mode produces `<frame-id>.png` or `<frame-id>.jpeg` is never stated.  
**Suggested fix:** Extend the `output-format=images` naming rule: *"`RGBA` and `GRAY` frames use `.png`; `RGB` frames use `.jpeg`."*

---

### W-06 — MEDIUM — `clip-shape=polygon` — clip polygon geometry undefined

**Sections:** §12.2  
**Gap:** §12.2 states `clip-shape` accepts `circle`, `square`, or `polygon`. For `circle` and `square`, the clip geometry is implied by the object's `width`/`height` bounding box. For `polygon`, there is no attribute or mechanism to supply the vertex list — no `clip-polygon-points` attribute or equivalent exists. An implementor cannot determine what polygon geometry to apply as the clip mask.  
**Suggested fix:** One of:
- (a) Document that `clip-shape=polygon` uses the object bounding box as a degenerate convex polygon (equivalent to `square`), or
- (b) Add a companion attribute `clip-polygon-points: <point-list>` to §12.2, or
- (c) Remove `polygon` from the valid `clip-shape` values and add it to §18.1 as an unsupported keyword.

### W-07 — MEDIUM — `background()` invalid parameter combinations — no §18 error entries

**Sections:** §7, §18  
**Gap:** §7 explains that the engine distinguishes background forms (solid / gradient / image) by key presence. The following cases are syntactically valid per the EBNF `<named-param-list>` but semantically invalid:
- `background()` — no parameters at all
- `background(color=black, color1=red)` — both solid and gradient keys mixed
- `background(color1=red)` — gradient with only `color1`, missing `color2`, `start`, and `end`

No §18 error covers any of these invalid parameter combinations. An implementor has no message guidance.  
**Suggested fix:** Add to §18.1 or §18.2:

| Condition | Message Pattern |
|---|---|
| `background()` with no parameters | `<file>:<line>: error: background: no parameters supplied` |
| `background()` with conflicting form keys (e.g., `color` and `color1` together) | `<file>:<line>: error: background: ambiguous parameter combination — cannot determine background form` |
| Gradient `background()` missing one or more of `color2`, `start`, `end` | `<file>:<line>: error: background: gradient form requires 'color1', 'color2', 'start', and 'end'` |

### W-08 — LOW — `start-cap` + `start-arrow` both specified — alias conflict precedence undefined

**Sections:** §9.3  
**Gap:** §9.3 documents `start-arrow` as an alias for `start-cap` (and `end-arrow` as an alias for `end-cap`). If a caller specifies both `start-cap=triangle` and `start-arrow=circle` in the same connector call, they are two distinct parameter keys (so the duplicate-key error in §18.1 does not trigger), but they refer to the same conceptual setting. Which value wins is undefined.  
**Suggested fix:** Add a note to §9.3: *"If both `start-cap` and `start-arrow` (or both `end-cap` and `end-arrow`) appear in the same call, a parse error is raised: `<file>:<line>: error: connector: 'start-cap' and 'start-arrow' are aliases and cannot both be specified`."*

### W-09 — LOW — `%` unit on scalar length parameters — reference dimension undefined

**Sections:** §15.3  
**Gap:** §15.3 defines `%` as "x%: of parent width; y%: of parent height." This is clear for coordinate pairs and bounding-box sizes. Scalar length parameters — `radius`, `line-width`, `corner-radius`, `blur-radius`, `pattern-length`, `pattern-gap`, `pattern-speed` — have no x/y axis. Which canvas dimension (width, height, or other) does `%` resolve against for these parameters is never stated.  
**Suggested fix:** Add to §15.3: *"When `%` is used on a scalar (non-positional) length parameter (e.g., `radius`, `line-width`, `corner-radius`), it resolves relative to the **canvas width**."* (Alternatively, state that `%` is not valid on scalar length parameters and produces a validation error.)

---

### W-10 — LOW — `hold-time=0` or negative `hold-time` — behavior undefined

**Sections:** §5, §18.2  
**Gap:** §5 documents `hold-time` with a default of `100` ms and notes "Decimal values are truncated to integer." `hold-time=0` would produce a 0 ms frame (meaning: advance as fast as possible, or potentially skip the frame entirely). A negative value (e.g., `hold-time=-100`) after truncation would be nonsensical. Neither case is addressed in §18.2.  
**Suggested fix:** Add to §18.2:

| Condition | Behaviour |
|---|---|
| `hold-time` value < 1 (after integer truncation) | Error halted: `<file>:<line>: error: 'hold-time' must be at least 1 ms, got <value>` |

---

### W-11 — LOW — Unit suffix on angle values — silently stripped or validation error?

**Sections:** §2.7, §8.0, §8.6, §8.7  
**Gap:** §2.7 states "Angle values are always degrees (no unit suffix needed)." However, angle parameters (`start-angle`, `end-angle`, `rotate`) accept `<value>` → `<length>` per the EBNF grammar, which means `start-angle=90px` is syntactically valid (it parses as a length of 90 px). Whether the engine silently strips the unit and treats the number as 90°, or raises a validation error for the unexpected unit, is never stated.  
**Suggested fix:** Add a note to §2.7 (or §8.0): *"Angle parameters (`rotate`, `start-angle`, `end-angle`) must be supplied as plain numbers without a unit suffix. Providing a unit suffix (e.g., `rotate=45px`) is a validation error: `<file>:<line>: error: angle parameter '<key>' does not accept a unit suffix`."*
