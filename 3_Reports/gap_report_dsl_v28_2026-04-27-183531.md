## Findings

### V-01 — HIGH — `colorspace` in both `<frame-attr>` and `<image-param>` — conflict precedence undefined

**Sections:** §3 EBNF, §5, §6  
**Gap:** The EBNF `<frame-attr>` includes `colorspace`, and `<image-param>` also includes `colorspace`. The `<frame>` rule allows frame attributes before *and* after the `image` statement, so both can appear in the same frame with different values:
```
begin_frame foo
  colorspace=RGBA           # frame-attr
  image width=100px; height=100px; colorspace=RGB;   # image-param
```
No rule states which takes precedence, whether a conflict is an error, or whether the two are treated as a single unified setting.  
**Suggested fix:** Add a note to §5 or §6: *"If `colorspace` appears in both the frame attributes and the `image` statement, the `image` statement value takes precedence. Specifying conflicting values emits a warning: `<file>:<line>: warning: colorspace '<frame-value>' in frame attributes overridden by '<image-value>' in image statement`."*

### V-02 — HIGH — Missing required parameters — no error entry in §18.1

**Sections:** §8–§11, §18.1  
**Gap:** Every primitive has Required=Yes parameters (e.g., `center` and `radius` on `circle`, `text` and `pos` on `font`, `src` and `pos` on `image(...)`). The §18.1 error table covers "Unrecognised parameter key" and "Function argument count mismatch" but has no entry for *missing* required parameters on a primitive call. An implementor has no guidance on the message to emit.  
**Suggested fix:** Add to §18.1:

| Error Condition | Message Pattern |
|---|---|
| Required parameter missing from a primitive call | `<file>:<line>: error: '<primitive>': required parameter '<key>' is missing` |

---

### V-03 — MEDIUM — `curved` connector with exactly 2 points — Catmull-Rom spline behavior undefined

**Sections:** §9.2  
**Gap:** §9.2 states `curved` uses a Catmull-Rom spline through all points. Catmull-Rom requires at least 3 control points to produce curvature; with exactly 2 points the spline is degenerate. Whether `curved` with 2 points degrades to a straight segment or is a validation error is not stated.  
**Suggested fix:** Add a note to §9.2: *"With exactly 2 points, `connector-type=curved` renders a single straight segment (equivalent to `straight`)."*

---

### V-04 — MEDIUM — `pie` and `arc` with `start-angle >= end-angle` — sweep direction and wrap-around undefined

**Sections:** §8.6, §8.7  
**Gap:** Angles outside 0–360 are normalised (§18.2). After normalisation, if `start-angle >= end-angle`, the sweep direction is ambiguous: does the arc go clockwise from start, wrapping through 360° to reach end (e.g., 270°→90° = 180° arc through 0°)? Or is it treated as a zero-angle arc? Or an error? This is entirely undocumented.  
**Suggested fix:** Add a note to §8.6 and §8.7:  
*"The arc is always drawn **clockwise** from `start-angle` to `end-angle`. If `start-angle >= end-angle` after normalisation, the arc sweeps clockwise through 360° (e.g., `start-angle=270, end-angle=90` produces a 180° arc passing through 0°). If `start-angle == end-angle` (after normalisation), nothing is rendered."*

### V-05 — MEDIUM — `RGB`/`RGBA` out-of-range component values — no error documented

**Sections:** §15.1, §3 EBNF `<byte>`, `<number-0-1>`  
**Gap:** The EBNF `<byte>` is defined as `[0-9]+` with a comment `(* 0–255 *)`, and `<number-0-1>` covers `0.0–1.0`. Both are only comment-level constraints — the grammar accepts `RGB(300, 0, 0)` or `RGBA(0,0,0,2.5)` syntactically. No §18 row covers out-of-range byte (>255) or out-of-range alpha (>1.0) values.  
**Suggested fix:** Add to §18.2:

| Condition | Behaviour |
|---|---|
| `RGB`/`RGBA` component `r`, `g`, or `b` outside 0–255 | Error halted: `<file>:<line>: error: color component '<n>' is out of range 0–255` |
| `RGBA` alpha value outside 0.0–1.0 | Error halted: `<file>:<line>: error: alpha value '<n>' is out of range 0.0–1.0` |

---

### V-06 — MEDIUM — `width=0` or `height=0` canvas — zero-dimension validity undefined

**Sections:** §6, §18.2  
**Gap:** §18.2 covers negative values for `radius`, `width`, `height` (error halted). But a zero canvas dimension (`width=0` or `height=0`) is not addressed. A zero-dimension canvas would cause `%`-unit calculations to produce `0` and could cause division-by-zero in gradient rendering. Whether this is silently allowed, clamped, or an error is not stated.  
**Suggested fix:** Extend the §18.2 negative-value row or add a new entry: *"`width` or `height` of zero in the `image` canvas statement is a validation error: `<file>:<line>: error: canvas 'width'/'height' must be greater than zero`."*

---

### V-07 — MEDIUM — `scale=0` — zero scale validity undefined

**Sections:** §8.0, §18.2  
**Gap:** §8.0 states negative `scale` is a validation error. §18.2 has the corresponding row. But `scale=0` (which collapses the element to a point, making it invisible) is neither explicitly allowed nor disallowed. An implementor does not know whether to render a zero-scale element as nothing or reject it as an error.  
**Suggested fix:** Add a note to §8.0: *"`scale=0` is valid and renders the element as invisible (zero-size). To suppress rendering, `scale=0` is the preferred idiom."* (Or: *"`scale=0` is a validation error — same rule as negative scale."*)

---

### V-08 — LOW — `pattern-color` default missing from §16.2 summary table

**Sections:** §9.6, §16.2  
**Gap:** §9.6 documents `pattern-color` with default "connector `color`". This default is absent from the §16.2 consolidated defaults table, which otherwise lists every connector parameter default.  
**Suggested fix:** Add to §16.2: `pattern-color` → `connector color`.

---

### V-09 — LOW — 3-digit hex shorthand `#RGB` — valid or parse error? Not stated

**Sections:** §15.1, §3 EBNF  
**Gap:** The EBNF `<color>` defines hex color as `'#' [0-9A-Fa-f]{6}` (exactly 6 digits). The common CSS shorthand `#RGB` (3 digits, each doubled) is widely used. Whether the DSL accepts `#FFF` as `#FFFFFF` or rejects it as a parse error is never stated.  
**Suggested fix:** Add a note to §15.1: *"Only 6-digit hex notation (`#RRGGBB`) is supported. 3-digit shorthand (`#RGB`) is not valid and produces an 'unrecognised color format' parse error."*

---

### V-10 — LOW — Gradient background with identical `start` and `end` points — undefined behavior

**Sections:** §7.2  
**Gap:** If `start` and `end` points in a gradient background are identical coordinates, the gradient direction vector is zero-length, causing a potential division-by-zero in the interpolation formula. No warning or fallback behavior is defined for this case.  
**Suggested fix:** Add to §7.2: *"If `start` and `end` are the same point, the entire canvas is filled with `color1`; a warning is emitted: `<file>:<line>: warning: gradient start and end are identical; rendering as solid color1`."*

---

### V-11 — LOW — Shadow offset direction convention not documented

**Sections:** §12.2  
**Gap:** The `shadow` attribute takes `offset-x offset-y blur-radius color`. The sign convention for offsets — whether positive `offset-x` shifts the shadow right and positive `offset-y` shifts it down — is never stated. The canvas coordinate system (§8.0.1: Y increases downward) implies it, but §12.2 contains no explicit statement.  
**Suggested fix:** Add to the §12.2 `shadow` description: *"Positive `offset-x` shifts the shadow rightward; positive `offset-y` shifts it downward, consistent with the canvas coordinate system (§8.0.1)."*
